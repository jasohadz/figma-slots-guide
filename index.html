Implementing Slots
in a Figma Library

Migrating to native slots isn't just swapping utilities for slot layers. It forces system architects to critically examine how components are composed, configured, and customized across the entire catalog.

Discovery
Audit existing assets, research patterns, explore alternatives before committing to a direction.

Decisions

Phase 03
Implementation
Convert component-by-component, from low- to high-order, with pilot testing before publish.
Phase 01

Discovery

Don't rush into implementation. Systematic research prevents architectural mistakes that are expensive to undo once they're published across a component catalog.

Audit — Document Each Component
  • Component name — canonical library name
  • Compositional role — Root, Header, Body, Footer, Actions, Start/End
  • Current slot utility usage — what's in use today
  • Variant-specific availability — does every variant support the slot?
  • Nested composable instances — subcomponents inside
  • All prop types — variants, text, boolean, nested
  • Props to keep vs. reconsider — triage during audit
Exploration — Build Before Deciding
  • Pattern-organized Figma file — one section per pattern
  • Multi-component examples — show pattern across 3+ components
  • Alternative approaches — hierarchy, layer structure, props, depth
  • Open questions log — track unresolved decisions with votes
  • Architecture decision notes — recommended direction per pattern
  • Default content variants — explore uniform vs. varying defaults
  • Code alignment check — match design explorations to code API
⌗
Examine Code Implementations
Review how slots are handled in your existing codebase across all platforms. Design slot structure should mirror the code API.
◎
Study Public Libraries
Analyze compositional patterns in GitHub Primer, MUI React, and similar libraries with strong composition traits.
◈
Use LLMs to Align
Use language models to reconcile design patterns with code patterns. Identify naming and structural discrepancies early.

!
Critical constraint: Never set padding on a slot layer. A component should not control the inset of its slot — that's the consumer's responsibility. Set direction, resizing behavior, and item spacing only.
i
Figma helpers vs. true API props: Distinguish between props that aid designer workflow (like a "Custom" boolean that activates slot mode) and props that reflect the actual code API. Document which is which.

Phase 03

Implementation

Convert the library component-by-component using a structured sequence. Move from low-order to high-order components, building confidence before tackling complex compositions.

01
One branch per component
Keep each conversion isolated. Avoid batching unrelated components in a single branch — it makes review and rollback harder. Branch hygiene
02
Start with low-order components
Begin with atomic components before composites. Foundations must be solid before building higher-order patterns on top. Ordering
03
Batch similar-pattern components
Group components that share the same slot approach (e.g. all repeating item groups, all components using items slot). Patterns transfer faster in batches. Efficiency
04
Defer complex nested structures
Postpone components with deep nesting until foundational patterns are proven. Examples: Alert with Text + Actions; Action List with Item, Start Visual, Text, End Visual. Complexity
05
Build test cases alongside assets
Construct test compositions in the same branch as the component. Seeing real usage surfaces layout issues before publish. Validation
06
Pilot test before full rollout
Publish a small cohort of components to willing teams first. Collect feedback, refine conventions, then proceed with the full catalog. Pilot
07
Coordinate on "configuration collapse"
Slot adoption naturally reduces configuration-heavy props in favor of composition. Align with consumers and engineers on which props are being retired and what replaces them. Alignment
